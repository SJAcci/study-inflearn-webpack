**** 프론트엔드 개발자를 위한 웹팩 ****

====== I. NPM
# npm 
> node package manager
> 자바스크립트 패키지 관리자
> 전세계 모든 자바스크립트 라이브러리를 관리

# npm 명령어
> 초기화 : npm init -y
> 설치 : npm install {target}
    > node_modules 하위에 라이브러리.js 가 설치

# 장단점
> 관리의 편의성, 라이브러리 의존성 관리가 용이, 라이브러리 설치의 편의성

# 전역 설치
> npm install {target} --global
> 시스템 레벨에 설치
> 어느 위치에서 해당 명령어를 실행했던지 간에 실행

# 지역 설치
> npm install {target} --save-prod : 배포용(dependencies)에 등록
> npm install {target} --save-dev (= npm i {target} -D): 개발용(devDependencies)에 등록
> 설치된 배포용 라이브러리는 npm run build로 빌드를 하면 최종 애플리케이션 코드 안에 포함 됨

# 웹팩
> 최신 프런트엔드 프레임워크에서 가장 많이 사용되는 모듈 번들러
> 웹 애플리케이션을 구성하는 자원들을 하나의 파일로 병합, 압축 해주는 동작을 모듈 번들링 이라 한다.
> 진입점은 하나, 나머지 연관 관계는 웹팩이 다 해석을 해줌
> bundle.js 에 필요한 내용이 다 들어있음
> 페이지 로딩 속도가 확연히 단축

# 웹팩 등장 배경
> 파일 단위의 자바스크립트 모듈 관리: 
    > 모듈 : 웹팩에서 지칭하는 모듈이라는 개념은 위와 같이 자바스크립트 모듈에만 국한되지 않고 웹 애플리케이션을 구성하는 모든 자원을 의미.
    > 일 단위로 변수를 관리
> 웹 개발 작업 자동화 도구 : HTML, CSS, JS 압축 / 이미지 압축 / CSS 전처리기 변환 등을 자동화 해주는 도구가 나타남 = gulp / grunt
> 웹 애플리케이션의 빠른 로딩 속도와 높은 성능 : 웹팩은 기본적으로 필요한 자원은 미리 로딩하는게 아니라 그 때 그 때 요청

# gulp
> 웹 자동화 도구
> 최적화 작업을 진행하기 위해서 사용
> 성능을 보장

# 웹팩으로 해결하려는 문제
> 웹팩에서 해결하고자 하는 기존의 문제점은 다음 4가지 입니다.
  > 자바스크립트 변수 유효 범위
  > 브라우저별 HTTP 요청 숫자의 제약
  > 사용하지 않는 코드의 관리
  > Dynamic Loading & Lazy Loading 미지원

# babel
> JS 최신 문법들을 최대한 많은 브라우저가 호환할 수 있게 변환해주는 도구

# import 와 export
> 변수 스코프 관리 및 모듈 시스템 구현 
> 모듈 차원에서의 관리
> 변수의 유효범위
> bundle.js 에서 모듈화된 것 합침

# Webpack 의 주요 역할
> 모듈화 된 여러 파일을 하나로 묶어주는 번들링

# source map
> 번들링 또는 압축된 JavaScript 코드를 디버깅할 때 원본 코드와 빌드된 코드 간의 연결 제공
